# Transport Layer
- Function Overview
    - Provides logical communication between application processes.
    - Reusing and Multiplexing
        - Reusing
            - Different applications on the sending side use the same transport layer protocol.
        - Multiplexing
            - The receiving end's transport layer can correctly deliver these data to the destination application process after stripping off the header.
    - Error Detection (Header and Data Parts)
    - TCP and UDP
- Addressing and Ports
    - Port Function
        - Allows each application in the application layer to forward its data down through the port.
        - Enables the transport layer to know which data to forward up through the port to the corresponding process of the application.
    - Port Number
        - 16-bit $$2^{16}=65536$$ different port numbers
        - Port division
            - Server side
                - Well-known ports
                    - 0 ~ 1023
                    - FTP
                        - 21
                    - TELNET
                        - 23
                    - SMTP
                        - 25
                    - DNS
                        - 53
                    - TFTP
                        - 69
                    - HTTP
                        - 80
                    - SNMP
                        - 161
                - Registered ports
                    - 1024 ~ 49151

            - Client side
                - Dynamically selected during client runtime
                - Also known as temporary ports
        - Sockets
            - Combines port and IP address
            - Socket = (IP Address: Port Number)
- UDP
    - Overview
        - Built on top of IP data packets
            - Reusing
            - Multiplexing
            - Error detection
        - No need to establish a connection
        - Stateless
        - Small header overhead
        - Does not guarantee reliable delivery
        - Message-oriented
        - 17 fields in the IP protocol
        - Packet length is determined by the sending program
    - Header Format
        - 8B four fields
            - Source port
                - Use when a response is needed
                - Can be all zeros if not required
            - Destination port
                - Must use when delivering the message to the endpoint
            - Length
                - Minimum of 8 (only header)
            - Checksum
                - Detects errors in the UDP data packet transmission.
                - Discards if an error occurs
                - The sending host does not calculate the checksum when filling with all zeros
    - Checksum
        - Adds a 12B pseudo-header before the UDP data packet
        - Pseudo-header is not the actual UDP header, used for checksum calculation.
- TCP
    - Overview
        - Connection-oriented
        - Reliable delivery
        - Full duplex communication
            - Both sides can send and receive data at any time
            - Two caches to temporarily store data
                - Sending cache
                    - Data passed from the sending program to the TCP sender.
                    - The data that has been sent but not yet received confirmation.
                - Receiving cache
                    - Data arriving in order but not yet read by the receiving application.
                    - Data arriving out of order
        - Byte stream oriented
            - Treated as a continuous byte stream only
        - Packet length determined by the receive window and current network congestion
    - Packets
        - Source port and destination port
            - 2B + 2B
        - Sequence number
            - 4B
            - 0 ~ $$2^{32}-1$$
            - Byte stream oriented
            - Each byte of the data stream being transmitted in a byte-stream is numbered sequentially.
            - The sequence number of the first byte of the data sent in this packet.
        - Acknowledgement number
            - 4B
            - Expected sequence number of the next data byte from the receiving party.
            - ACK=N means that all data up to sequence N-1 has been received correctly.
        - Data offset
            - Represents header length
            - Indicates how far the data field of the TCP packet starts from the beginning of the TCP packet.
        - Reserved
            - Occupies 6 bits
            - Future use
            - Currently set to 0
        - Urgent bit URG
            - If URG=1
                - Indicates that the urgent pointer field is valid.
                - Should be delivered as soon as possible.
        - Acknowledgement bit ACK
            - If ACK=1
                - The acknowledgement number field is valid.
            - If ACK=0
                - Invalid acknowledgement number
        - Push bit PSH
            - When the TCP receiving end receives a packet with PSH=1.
            - Delivers to the receiving application process as soon as possible.
            - Not waiting for the cache to be full before delivering.
        - Reset bit RST
            - If RST=1
                - Indicates that there is a serious error in the TCP connection.
                - Must release the link.
                - Reestablish connection
        - Synchronization bit SYN
            - If SYN=1 indicates a connection request or an accepted connection.
            - SYN=1 ACK=0 indicates a connection request message.
                The other side agrees to establish a connection.
                Then use SYN=1 ACK=1 in the response message.
        - Fin bit FIN
            - Used to release a connection
            - FIN=1 indicates that the sender has finished sending its data and wants to release the transport link.
    - Window
        - 2B
        - $$2^{16}-1$$
        - The amount of data that the other party is allowed to send
        - Set confirmation number to 701, window field to 1000
            - Any byte from 701 to 1700
        - Checksum
            - 2B
            - The checksum field covers both header and data.
        - Urgent pointer
            - 2B
            - If URG=1, the urgent pointer is valid.
            - Should be delivered as soon as possible.
        - Options
            - Variable length
            - Maximum segment size MSS
                - The maximum data field length of a TCP packet.
        - Padding
            - To ensure that the entire header length is a multiple of 4B.
    - Connection Management
        - Three-way handshake
            - Establish connection
            - Data transfer
            - Release connection
        - Client/Server mode
        - Establishing a connection
            - Three-way handshake
            - Before connecting, the server process enters LISTEN and waits for the client's connection request.
            - The client sends a SYN=1 seq=x message to the server TCP.
                SYN packet cannot carry data.
                Consumes one sequence number.
                The client enters SYN-SENT state.
            - The server receives the connection request message and agrees to connect.
                Sends an acknowledgement message with SYN=1 ACK=1 seq=y ack=x+1.
                The acknowledgement message cannot carry data.
                Consumes one sequence number.
                The server enters SYN-RCVD state.
            - The client receives the confirmation message and sends another confirmation to the server.
                ACK=1 seq=x+1 ack=y+1
                This packet can carry data.
                Does not consume sequence numbers.
                The client enters ESTABLISHED state.
            - 3-way handshake is susceptible to SYN flood attacks.
        - Disconnecting a connection
            - Four-way handshake
            - The client intends to disconnect sends the TCP connection release message.
                - Actively closes the TCP connection.
                - FIN=1 seq=u
                - The sequence number is the last byte of data transmitted previously.
                - The client enters FIN-WAIT-1 state.
            - The server receives the connection release message.
                - ACK=1 seq=y ack=u+1
                - The sequence number equals the first byte of the last transmitted data.
                - The server enters CLOSE-WAIT.
                - The server may still send data to the client.
                - The client must continue to receive it.
            - The server sends a FIN message to the client to release the connection.
                - FIN=1 ACK=1 seq=w ack=u+1
                - If there is partial data transmission, the sequence number is w.
                - The sequence number of the acknowledgement number is u+1
                - The server enters LAST-ACK state
            - The client receives the confirmation message and sends a final ACK to complete the disconnection.
                - ACK=1 seq=u+1 ack=w+1
                - At this point, the TCP connection has not yet been released.
                - There must be at least 2MSL (maximum segment lifetime) time delay before the connection is actually closed.
                - The client enters CLOSED state
